1 Overview & Strategic Objectives 

The WorkSphere HR Module elevates the existing application from a task and attendance tracker into a full-scale Human Capital Management (HCM) system. It serves as the central hub for the complete employee lifecycle — from onboarding and compensation through performance, asset management, grievance resolution, and finally offboarding. 

This document supersedes the original blueprint and covers all missing specifications: audit logging, notification architecture, offboarding workflows, DTO contracts, tax calculation logic, transactional safety, and frontend route guarding. 

1.1 Design Principles 

* 
**Data integrity first** — every destructive or financial operation must be transactional and auditable. 


* 
**Progressive complexity** — three clearly separated implementation phases, with payroll gated behind stable underlying data. 


* 
**Role-enforced access** — no endpoint or UI screen is reachable without the correct clearance level. 


* 
**Failure safety** — payroll generation failures must roll back atomically; no partial records are ever persisted. 


* 
**Notification-driven** — every state change in payroll, appraisals, tickets, and assets triggers a user-facing notification.

2 Hierarchy, Authority & Clearance Levels 

The HR Module inherits and extends the existing Role and Employee.manager authority structure. A new `HR_MANAGER` system role is introduced at clearance Level 2, sitting between `SUPER_ADMIN` and team-level `MANAGER`s. 

| Role | Clearance | Key Capabilities |
| --- | --- | --- |
| **SUPER_ADMIN / SYSTEM_ADMIN** | Level 1 | Full configuration, deletion of sensitive records, payroll overrides, system-wide audit access, policy management. 

 |
| **HR_MANAGER** | Level 2 | Org-wide payroll generation, asset management, all appraisals, offboarding initiation, grievance resolution, document management. 

 |
| **MANAGER** | Level 3 | Appraise direct reports, view team compensation summaries, initiate promotion/transfer requests, assign team assets. 

 |
| **EMPLOYEE** | Level 4 | View own payslips, submit grievances, acknowledge appraisals, view assigned assets, raise helpdesk tickets. 

 |

2.1 Role Escalation Rules 

* An `HR_MANAGER` cannot modify their own payroll record. A second `HR_MANAGER` or `SUPER_ADMIN` must approve. 


* A `MANAGER` can only appraise employees where `Employee.manager == self`. Cross-team appraisals are forbidden. 


* `EMPLOYEE` self-review comments are append-only. Once submitted, they cannot be edited. 


* Any role change (e.g., `EMPLOYEE` to `MANAGER`) must be logged in `AuditLog` with the approving `HR_MANAGER`'s ID.

4 Feature Specifications 

4.1 Payroll & Compensation 

**4.1.1 Salary Structure Management** 

* HR_MANAGER can define SalaryStructures at the JobPosition level (default) or override for a specific Employee. 


* Structures are versioned by effectiveDate. The most recent past-dated structure is always used for calculation. 


* Changes to a SalaryStructure do not retroactively alter already-PROCESSED PayrollRecords. 



**4.1.2 Monthly Payroll Generation** 

* Triggered manually by HR_MANAGER or automatically by MonthlyPayrollDraftJob (cron: 0 0 9 25 * ?) on the 25th. 


* PayrollCalculationService fetches attendance records and counts lopDays. 


* LOP Formula: `lopDeduction = (grossPay / workingDays) * lopDays` 


* Tax is calculated via TaxCalculationService (see Section 5.3). 


* All calculations happen inside a single `@Transactional` block. Failure rolls back all records for that run. 


* Generated records start in DRAFT status. HR reviews and promotes to PROCESSED, then triggers payment to mark PAID. 



**4.1.3 Payslip PDF Generation** 

* Triggered when a PayrollRecord is moved to PROCESSED status. 


* Generated server-side using the Jasper Reports or Flying Saucer (HTML-to-PDF) library. 


* PDF stored at `/payslips/{year}/{month}/{employee_id}.pdf` or uploaded to S3. 


* Employees are notified via Notification (type: PAYSLIP_READY) with a secure download link. 


* Payslips are never regenerated once PAID. HR must contact SUPER_ADMIN for amendments. 



4.2 Performance Appraisals 

* HR_MANAGER creates a review cycle, setting cycleType, reviewPeriod, and dueDate. 


* System auto-generates PerformanceAppraisal records for all active employees, linked to their manager as reviewer. 


* AppraisalService queries TaskHistoryRepository to populate taskCompletionRate, tasksAssigned, tasksCompletedOnTime. 


* MANAGER submits scores and comments -> status moves to REVIEWED -> Employee notification sent. 


* Employee reads appraisal, adds optional self-review comment -> Acknowledges -> status moves to ACKNOWLEDGED. 


* ACKNOWLEDGED appraisals are locked. No further edits are possible by any role except SUPER_ADMIN. 



4.3 Asset Management 

* HR_MANAGER or SUPER_ADMIN can create, edit, and retire CompanyAsset records. 


* Asset assignment requires recording assignedAt and updating AuditLog. 


* Employee receives an ASSET_ASSIGNED notification on allocation. 


* During offboarding, AssetManagementService checks all assets assigned to the departing employee and sets OffboardingRecord.assetsCleared = true only when all are returned. 


* An asset returned in a DAMAGED condition auto-creates a GrievanceTicket of category IT_SUPPORT for documentation. 



4.4 Helpdesk & Grievance Ticketing 

* Any Employee can raise a GrievanceTicket with a category, priority, subject, and description. 


* ticketNumber is auto-generated: GR-{YEAR}-{4-digit-sequence} e.g. GR-2026-0001. 


* HR_MANAGER assigns the ticket to a handler; status moves from OPEN to IN_PROGRESS. 


* Threaded comments (TicketComment) allow back-and-forth communication. 


* Internal comments (isInternal: true) are hidden from the employee submitter. 


* On resolution, HR populates the resolution field and sets status to RESOLVED. Employee receives a notification. 



4.5 Offboarding Workflow (NEW) 

* HR_MANAGER initiates offboarding: OffboardingRecord created with status INITIATED. 


* System auto-assigns three parallel checklist tasks: Asset Return, Final Payroll, Access Revocation. 


* AssetManagementService: notifies employee to return all assigned assets. 


* IT (via ticket): access revocation ticket auto-created and assigned to IT_SUPPORT category. 


* PayrollCalculationService: generates a final PayrollRecord for the partial month on lastWorkingDate. 


* Once all three checklist items are true, HR marks status COMPLETED and deactivates the Employee record. 


* AuditLog entries are written at each step. Complete offboarding history is retained permanently.

5.2 Service Layer Architecture 

**PayrollCalculationService** 

* Injects: `AttendanceRepository`, `LeaveTransactionRepository`, `SalaryStructureRepository`, `TaxCalculationService`. 


* 
`generateForMonth(int month, int year)`: iterates all active employees, calls `calculateForEmployee()` per employee. 


* 
`calculateForEmployee()`: computes grossPay, queries lopDays, calls `TaxCalculationService`, returns PayrollRecord (DRAFT). 


* The entire `generateForMonth()` is annotated `@Transactional(rollbackFor = Exception.class)`. Any exception rolls back all saved records for that run. 


* If a PayrollRecord for `(employee, month, year)` already exists and is DRAFT, it is recalculated. If PROCESSED or PAID, it throws a `PayrollAlreadyProcessedException`. 



**AuditService** 

* Called by every service method that mutates sensitive data. 


* 
`log(entityType, entityId, action, performedBy, prev, next)`: persists a new AuditLog entry. 


* Uses a separate `@Transactional(propagation = REQUIRES_NEW)` to ensure audit logs are written even if the calling transaction rolls back. 



**NotificationService** 

* 
`send(recipientId, type, title, message, refId, refType)`: persists an in-app Notification. 


* Also triggers an async email dispatch via `JavaMailSender` using Thymeleaf HTML templates. 


* Email dispatch is `@Async` to avoid blocking the calling request thread. 



**OffboardingService** 

* 
`initiateOffboarding()`: creates OffboardingRecord, triggers asset return request notifications, auto-creates IT access revocation ticket. 


* 
`checkAndComplete()`: called after each checklist item update; if all three flags are true, deactivates Employee and sends confirmation notification. 



5.3 Tax Calculation Logic (NEW — Previously Missing) 

Tax (TDS — Tax Deducted at Source) is computed on a projected annual income basis and deducted equally each month. 

**Algorithm:** 

1. Compute `projectedAnnualGross = currentMonthGross * 12`. 


2. Subtract standard deduction (currently INR 50,000 per Indian IT law). 


3. Subtract annual PF contribution (employee share). 


4. Apply progressive tax slabs to the resulting taxableIncome. 


5. Divide annualTax by 12 to get monthlyTaxDeduction. 



**Tax Slab Configuration (India — New Regime as of FY 2025-26)** 
| Income Slab | Tax Rate | Notes |
| :--- | :--- | :--- |
| Up to INR 3,00,000 | 0% | Exempt  |
| INR 3,00,001 – 7,00,000 | 5% | Rebate u/s 87A up to INR 25,000  |
| INR 7,00,001 – 10,00,000 | 10% |   |
| INR 10,00,001 – 12,00,000 | 15% |   |
| INR 12,00,001 – 15,00,000 | 20% |   |
| Above INR 15,00,000 | 30% | Plus 4% Health & Education Cess  |

Tax slab values must be stored in a `TaxSlabConfig` table, not hardcoded. HR_MANAGER can update them each financial year. If the employee declares HRA exemptions, investments under 80C, or medical allowances, these must be submitted via an `EmployeeTaxDeclaration` form before the payroll run. In the absence of a declaration, the system uses zero exemptions (worst-case TDS).

All HR routes are wrapped in `ProtectedRoute` components that check the authenticated user's role before rendering. Unauthorized access redirects to `/unauthorized`. 

**Implementation Pattern:** 

*

* Routes are defined in `AppRouter.jsx` using React Router v6 nested routes with the `ProtectedRoute` wrapper. 


* Frontend guards are UX conveniences only. Backend `@PreAuthorize` annotations are the true enforcement layer.

9 Security, Error Handling & Transactional Safety 

9.1 Backend Security 

* Every controller method is annotated with `@PreAuthorize("hasRole('HR_MANAGER')")` or equivalent. Frontend guards are convenience only. 


* Employees accessing their own records are validated via `SecurityContextHolder` — the authenticated user's ID must match the requested employee ID. 


* Payroll endpoints are additionally protected: `HR_MANAGER` cannot modify their own `PayrollRecord` (service-level check). 


* All sensitive endpoints log to `AuditLog` before returning. Failed attempts are logged with action: `REJECTED`. 


* API responses never return internal stack traces. A `GlobalExceptionHandler` maps exceptions to standardized `ErrorResponse` DTOs. 



9.2 Transactional Safety for Payroll 

* `PayrollCalculationService.generateForMonth()` is `@Transactional(rollbackFor = Exception.class)`. If any single employee's calculation fails, the entire batch rolls back. 


* Payroll runs are idempotent for DRAFT records — re-running recalculates them. PROCESSED/PAID records are immutable. 


* `PayrollRecord` status transitions follow a strict machine: DRAFT -> PROCESSED -> PAID. Reverse transitions are forbidden. 


* A `PayrollRunLog` entity (optional Phase 3 addition) can record the start time, end time, record count, and error messages for each payroll run for audit purposes. 



9.3 Data Validation 

* All request DTOs use Jakarta Bean Validation (`@NotNull`, `@Size`, `@Min`, `@Max`, `@FutureOrPresent`). 


* A `@ControllerAdvice` `GlobalExceptionHandler` catches `MethodArgumentNotValidException` and returns a structured list of field errors. 


* Business-level validations (e.g., duplicate payroll, missing salary structure) throw custom checked exceptions: `PayrollAlreadyProcessedException`, `SalaryStructureNotFoundException`.
